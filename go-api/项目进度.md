# 项目进度

## 1. 文档用途

本文件是本项目重构与改造的唯一进度面板，用于统一记录：

- 当前阶段与完成状态
- 本阶段任务清单
- 已完成改动摘要
- 下一步计划
- 风险、阻塞与注意事项

原则：

- 任何会修改仓库内容的变更，都必须同步更新本文件。
- 不允许只改代码、不记进度。
- 每次开始新一轮代码改动前，先写计划；完成后立刻回写结果。

## 2. 强制更新约定

每次改动仓库内容后，至少同步更新：

1. `阶段总览`
2. `当前执行计划`
3. `当前状态`
4. `下一步`
5. `最近变更记录`

状态约定：

- `未开始`：还没有落地任何文件变更
- `进行中`：已经开始落地，但未完成该阶段目标
- `已完成`：该阶段目标已完成，并且已完成基本验证
- `阻塞`：无法继续，需先解决外部依赖或决策问题

## 3. 总目标

在不破坏现有业务路径、接口协议和 PHP 兼容能力的前提下，逐步把当前单体 Go 服务重构为：

- 更清晰的装配层
- 可维护的路由组织
- 可管理的后台任务与生命周期
- 更稳定的业务域边界
- 更容易测试和持续演进的代码结构

明确约束：

- 默认不改公开 API 路径；如需调整，必须先补兼容层或迁移说明
- 默认不改 HTTP 方法；兼容入口优先保留原有挂载方式
- 默认不改响应结构；修复性调整应尽量保持字段语义与兼容输出不变
- 默认不主动改 SQL 语义；除非为了修复已确认缺陷或消除明显重复逻辑，否则不调整查询结果、过滤条件、排序和写入副作用
- 默认不改 PHP 兼容协议；兼容桥仍作为硬约束保留

## 7. 风险与阻塞

当前已知风险：

- 项目整体分层方向已经稳定，但仍存在“新结构已建立，旧兼容宿主未完全退场”的迁移期特征；阅读成本主要集中在 `internal/service`、`legacy/*`、`pluginruntime/*` 的交界处
- `internal/service` 当前已被定义为兼容宿主目录，不再是新业务入口；但目录体量仍大，且文件名虽然已加前缀，认知负担仍明显高于 `modules/*`
- 应用装配和生命周期已经从 `main.go` 中拆出，但依赖注入还未真正收紧，`database.DB`、`cache.RDB`、`config.Global`、`license.Global` 这类全局访问仍是主要耦合来源
- `internal/routes/register.go` 已完成主业务 / 插件系统分组，但仍是单点总装配入口；继续扩展时需要警惕把更多模块耦合堆回这里
- 数据访问边界只完成了部分收口；`shared/db` 已承接一部分查询能力，但高价值写路径里的 SQL、事务和副作用仍分散在模块、兼容宿主和运行态工具之间
- PHP 兼容桥、旧 OpenAPI 与插件生态仍是硬约束，不能假设可以直接用“纯 Go、纯新架构”一次性替换
- 复杂业务链路测试仍不够，当前更偏向边界守卫、handler 回归和伪 DB 集成；还不是完整端到端覆盖
- 本环境下 `go build` / `go test` 结束时可能因 trim cache 报 `Access is denied`；当前以编译/测试主体是否完成为准
- 明文密码兼容当前仍是迁移期开关，彻底关闭前需要确认生产数据状态
- 认证与后台高权限工具仍有安全治理空间，例如 refresh/access token 边界、query 参数传 token/key、后台下载并安装外部 CLI 的供应链风险

### 当前确认的 5 个结构性问题（2026-03-14 复盘）

1. `internal/service` 的语义虽然已经被重新定义，但目录体量仍然过大。
   当前它更像“兼容宿主收容区”而不是业务层，这个方向是对的；问题在于目录仍然承载大量历史上下文，新人依然要先学会区分 `compat_* / plugin_host_* / shared_*` 才能安全修改。
2. 依赖注入没有真正成为默认路径。
   `BuildApp` 已提供应用级装配，但运行时和业务代码仍大量回落到全局变量；这会削弱模块边界，也让测试和生命周期治理继续依赖隐式约定。
3. 路由入口仍然偏集中。
   `internal/routes/register.go` 已比过去清晰很多，但它仍然承担整站拓扑索引角色；继续加模块时如果缺少子装配层，认知压力会再次上升。
4. 兼容层已经隔离，但还没有完全“瘦成适配器”。
   `legacy/*` 与部分插件运行桥仍保留实际业务判断和运行态逻辑；这意味着结构上虽然分了目录，但部分复杂度还没有真正从兼容层退出。
5. 当前剩余问题的主轴已经不是“继续拆文件”，而是“稳定治理”。
   继续机械拆分目录的收益已经很低，后续更应该集中在边界约束、全局依赖收口、数据访问边界、认证安全和高价值回归测试。

## 8. 下一步

结构演进原则：

- 当前阶段不再以“继续拆文件”为主目标，而是以“让现有结构长期可维护”为目标。
- 继续保留兼容宿主与插件运行桥，但必须把它们视为受控边界，而不是新的通用落点。
- 优先收紧依赖方向和目录规则，再考虑是否继续搬迁个别宿主实现。

下一步默认执行顺序：

1. 先固化目录规则：新业务只能进 `modules/*`，新兼容协议逻辑只能进 `legacy/*`，新运行态能力只能进 `pluginruntime/* / autosync/* / dbtools/* / runtimeops/*`
2. 再收紧依赖注入：新增代码禁止继续扩大 `database.DB / cache.RDB / config.Global / license.Global` 的直接使用面，逐步改成显式注入
3. 然后拆分路由总装配：把 `routes/register.go` 继续按 `core / legacy / plugins` 做子装配，降低单入口负担
4. 再梳理高价值写路径：优先处理 `order / tenant / admin / user` 的 SQL、事务和副作用边界，而不是继续机械迁目录
5. 最后做稳定治理：补齐认证安全、兼容入口回归、后台运维工具链和失败分支测试

建议的实际推进顺序：

1. 先处理结构治理规则
   包括 `internal/service` 禁止承接新业务、`modules/*` 边界守卫延续、运行态目录职责文档保持同步。
2. 再处理全局依赖与装配层
   优先从新增或高频改动代码开始，把依赖显式下传，避免继续扩大隐式共享状态。
3. 然后处理安全与运维高风险点
   重点是认证边界、query token/key、后台下载外部 CLI、运行态权限和外部调用。
4. 最后补高价值回归测试
   围绕 `auth / tenant / user / order / admin ops / legacy` 做链路补强，把现在的结构收益固定下来。
5. 继续识别并压缩“还能自圆其说，但已经偏薄”的运行态薄包
   当前优先观察 `admin` 模块内局部 glue 是否需要继续按 owner 拆小，而不是再新增顶级聚合包。

## 当前结论

- 结构性重构主线可以判定为完成：`main / bootstrap / routes / modules / legacy / pluginruntime` 的目录职责已经基本成形，当前不再是“架子没搭起来”的问题。
- 当前结构的主要短板不在目录数量，而在迁移期残留的认知成本：`internal/service` 仍然偏大、全局依赖仍然普遍、总路由装配仍然集中、兼容层还没完全瘦成窄适配器。
- 当前不建议继续以“删掉 `service`”或“继续大规模搬文件”为目标推进。后续更应该转向结构治理、依赖收口、安全治理和高价值回归测试。

## 下一阶段建议

1. 目录规则治理：继续把 `internal/service` 固定为兼容宿主目录，明确禁止承接新业务实现。
2. 依赖治理：逐步压缩 `database.DB / cache.RDB / config.Global / license.Global` 这类全局访问，让装配层真正成为依赖入口。
3. 路由治理：为 `routes/register.go` 再加一层 `core / legacy / plugins` 子装配，避免总入口继续膨胀。
4. 安全治理：优先处理 refresh/access token 边界、query token/key、后台外部 CLI 安装和兼容桥鉴权。
5. 回归测试治理：继续围绕 `auth / tenant / user / order / admin ops / legacy` 增加高价值链路测试，而不是只补参数校验型测试。
